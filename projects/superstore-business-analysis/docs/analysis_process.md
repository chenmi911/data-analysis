# 分析过程说明

## 1. 为什么先用 pandas 做清洗

Superstore 原始数据是一张订单明细表，字段包括订单、客户、地区、品类、销售额、数量、折扣和利润。这个项目的目标不是直接在原始 CSV 上写 SQL，而是先把数据整理成一张适合分析的标准表。

Python 这一层只做数据准备：

| 动作 | 目的 |
|---|---|
| 使用 `cp1252` 读取 CSV | 解决原始文件不是 UTF-8 导致的读取错误 |
| 字段名改成 snake_case | 避免 SQL 中反复使用带空格和连字符的字段名 |
| 转换 `order_date`、`ship_date` | 后续才能计算月份、发货天数和客户最近购买时间 |
| 新增 `ship_days` | 保留基础履约时效字段 |
| 新增 `profit_margin` | 方便后续判断销售额和利润是否匹配 |

这个分工是为了让 Python 负责清洗，MySQL 负责分析。后面的 SQL 不再直接处理原始字段名，也不再重新解析日期。

## 2. 为什么异常行要剔除并留痕

数据清洗不是只把问题 `print` 出来。这个项目中先检查以下问题：

```text
缺失关键字段
ship_date < order_date
sales <= 0
quantity <= 0
discount < 0 或 discount > 1
重复行
```

如果发现这些问题，脚本会把异常行输出到：

```text
data/superstore_rejected_rows.csv
```

同时只把通过规则的数据写入：

```text
data/superstore_orders_clean.csv
```

这次实际数据中异常行数为 0，所以清洗后仍然是 9994 行。即使没有异常，代码里也保留处理策略，避免面试或复盘时只能说“检查了但没处理”。

## 3. 为什么 SQL 先做经营总览

导入 MySQL 后，第一步不是直接看细分品类，而是先确认整体经营规模和趋势：

| 指标 | 口径 |
|---|---|
| 销售额 | `SUM(sales)` |
| 利润 | `SUM(profit)` |
| 利润率 | `SUM(profit) / SUM(sales)` |
| 订单数 | `COUNT(DISTINCT order_id)` |
| 客单价 | `SUM(sales) / COUNT(DISTINCT order_id)` |

这个顺序对应真实经营分析：先回答公司整体有没有增长，再判断增长质量是否可靠。如果只看销售额，容易忽略利润率下降或亏损月份。

月度分析里使用 `LAG()` 计算环比，是为了让趋势分析不只是列出每个月结果，而是能识别变化幅度：

```sql
LAG(total_sales) OVER (ORDER BY order_month)
```

这类写法对应企业里的 GMV、利润、订单数、活跃用户等月度经营看板。

## 4. 为什么再看区域和品类

整体利润为正，并不代表每个业务结构都健康。Superstore 数据里有 `region`、`category`、`sub_category`，所以第二层分析按区域和品类拆解。

这一步重点不是找销售额最高的对象，而是区分：

| 情况 | 含义 |
|---|---|
| 销售额高，利润也高 | 核心贡献项 |
| 销售额高，利润率低 | 规模大但质量差 |
| 总利润为负 | 需要优先诊断 |
| 销售额低，利润率高 | 可能是小而优的机会项 |

所以 SQL 中子品类分析按利润排序，而不是只按销售额排序。这样能发现 `Tables`、`Bookcases`、`Supplies` 这类利润问题，而不会被销售规模掩盖。

## 5. 为什么单独分析折扣

折扣是这个数据集中最接近“业务动作”的字段。销售额、利润和折扣放在一起看，可以回答一个更接近管理层的问题：

```text
促销有没有带来利润，还是只带来了销售额？
```

SQL 使用 `CASE WHEN` 把折扣分成区间：

```text
0
0-10%
10%-20%
20%-30%
>30%
```

再计算每个区间的销售额、利润、利润率和亏损率。这个分析链条比单纯 `GROUP BY discount` 更适合汇报，因为区间结果更容易解释，也更接近业务上设置折扣政策的方式。

高折扣子品类单独查询，是为了找出需要动作的对象。例如高折扣下亏损最严重的子品类，可以作为折扣审批或促销复盘的重点清单。

## 6. 为什么要补客户维度和 RFM

只看区域和品类还不够，因为 Superstore 数据里有客户 ID 和客户名称。客户维度能回答：

```text
谁贡献了利润？
谁消费金额高但实际亏损？
哪些客户值得运营？
哪些客户可能已经流失？
```

RFM 使用三个指标：

| 指标 | 本项目口径 | 方向 |
|---|---|---|
| R | 距最近一次下单过去多少天 | 越小越好 |
| F | 去重订单数 | 越大越好 |
| M | 销售额总和 | 越大越好 |

SQL 中用 `NTILE(5)` 给 F 和 M 打 1-5 分。R 的方向相反，所以写成：

```sql
6 - NTILE(5) OVER (ORDER BY recency_days)
```

这样最近购买的客户 R 分更高。最后用 `CASE WHEN` 把客户分成高价值客户、重点发展客户、新近低频客户、流失风险客户、高消费低利润客户等类型。

RFM 的价值不是模型多复杂，而是把客户运营从“看总销售额”推进到“看最近购买、购买频次、消费金额和利润质量”。

## 7. 为什么结论里要写数据限制

这个数据集没有成本明细、物流费用、库存、营销活动、广告投放、客户访问和注册数据。因此当前分析不能直接回答：

```text
真实毛利是多少？
库存周转好不好？
广告 ROI 高不高？
客户从哪里来？
促销是否提高了长期留存？
```

所以结论只能写成经营诊断和线索识别，不能写成完整经营归因。更稳妥的表达是：

- 高折扣订单利润风险明显。
- 部分子品类拖累整体利润。
- 高消费客户不一定是高价值客户。
- 后续需要补充成本、库存和营销数据再做深入判断。

## 8. 企业分析中的对应能力

| 本项目动作 | 企业常见场景 |
|---|---|
| 订单明细清洗 | 交易表、销售明细、CRM 明细整理 |
| 异常行剔除并留痕 | 数据质量监控、问题数据追踪 |
| 年度/月度经营总览 | 周报、月报、经营看板 |
| 环比分析 | GMV、利润、订单量、活跃用户趋势判断 |
| 区域和品类诊断 | 门店、地区、商品线表现拆解 |
| 折扣分桶 | 促销复盘、价格策略评估 |
| 高折扣亏损识别 | 审批规则、风险订单监控 |
| RFM 分层 | 客户运营、会员分层、流失预警 |

这个项目更适合定位为零售经营分析和 SQL 业务分析项目。它的重点不是复杂建模，而是指标口径清楚、分析顺序合理、结论能落到业务动作。
